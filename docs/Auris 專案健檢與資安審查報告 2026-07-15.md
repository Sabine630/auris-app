## 2. 高優先：備份還原不是原子操作，失敗時會留下半套資料

### 證據位置

- `auris-vue/src/services/db.js:101-143`
- `auris-vue/src/views/SettingsView.vue:291-318`

### 問題 A：缺少 store 的備份仍會通過

目前驗證邏輯：

```js
const rows = jsonData.data[s];
if (rows == null) continue;
```

但驗證完成後會清空所有 store：

```js
for (const s of ALL_STORES) {
  await dbClear(s);
}
```

因此備份若缺少 `messages`，仍會被接受，但既有 `messages` 會被清空且不會還原。

### 問題 B：清空與還原分散在多個 transaction

`dbClear()` 與每次 `dbPut()` 都各自建立一個 IndexedDB transaction。若還原中途發生 quota、I/O 或資料寫入錯誤，先前的清空與部分寫入不會 rollback。

### 已重現結果

使用實際 `db.js` 搭配一次性 IndexedDB 替身測試：

1. 只含 `characters`、`settings` 的備份被判定匯入成功。
2. 原有 `messages` 被清空。
3. 模擬第二筆訊息發生 `QuotaExceededError`：
   - 新角色已寫入
   - 第一筆新訊息已寫入
   - 第二筆失敗
   - 舊資料已不存在
   - 原 API key 尚未走到補回步驟，結果為空
4. UI 只顯示「匯入失敗：檔案格式錯誤或損毀」，但資料庫已被部分改寫。

### 風險

- 使用者目前資料可能被部分清空
- App 重新載入後可能進入不一致狀態
- 備份檔雖仍存在，可重試，但目的端原資料已遭破壞

### 修復建議

1. 以單一 multi-store `readwrite` transaction 完成「清空＋還原」。
2. 任一筆失敗時 abort，讓 IndexedDB rollback 全部異動。
3. 備份版本需定義必填 store manifest；舊版本透過明確 migration 補空陣列，不可對任意缺失靜默 `continue`。
4. 驗證每個 store 的必要欄位與基本型別，不只檢查 `id`／`key` 是否存在。
5. 匯入前可先估算資料量與 Storage quota。
6. 錯誤訊息需區分格式錯誤、quota、transaction abort 與未知錯誤。

### 驗收條件

- [ ] 缺少必填 store 的同版備份會在清空前被拒絕
- [ ] 舊版備份透過 migration 明確補齊 store
- [ ] 模擬第 N 筆 `put` 失敗後，目的端所有原資料完整保留
- [ ] 匯入成功後所有 store 與備份內容一致
- [ ] 原 key 的保留／清除策略有獨立測試
- [ ] 為 `exportAllData()`、`importAllData()` 新增完整單元測試

## 3. 中優先：備份可改寫 API endpoint，再搭配目的端既有 key

### 證據位置

- `auris-vue/src/services/db.js:85-143`
- `auris-vue/src/services/llm.js:24-29`
- `auris-vue/src/services/llm.js:225-257`

### 精確適用條件

備份檔本身不含 API key。風險只在以下條件同時成立時存在：

1. 在同一 origin 匯入
2. 目的端匯入前已存有 API key
3. 匯入不可信或遭修改的備份
4. 備份指定新的 `api_base`／`api_provider`

跨正式站與 Vercel、新裝置、新瀏覽器，或目的端原本沒有 key 時，不成立。

### 已重現結果

- 匯出資料中沒有 `api_key`
- 匯入含 `api_base: https://evil.example/v1` 的備份
- 匯入後目的端原 key 被保留
- `api_base` 已變成備份指定值

### 修復建議

- 匯入時忽略 `api_key`、`api_base`、`api_provider`、`api_model`
- 或將 API 設定視為一組不可拆分的本機設定
- 若 endpoint／provider 改變，清除現有 key 並要求使用者重新輸入
- 自訂 endpoint 至少驗證為合法 HTTPS URL，並在儲存前顯示目標 hostname

### 驗收條件

- [ ] 匯入備份無法靜默改變現有 key 的傳送目的地
- [ ] endpoint 改變後，未重新授權前不會發送背景 AI 請求
- [ ] 同 origin、跨 origin、目的端無 key 三種情境都有測試

## 4. 中優先：Vertex token 快取未綁定 service account

### 證據位置

- `auris-vue/src/services/api.js:25-71`

目前只有：

```js
let _vtok = null, _vtokExp = 0;
```

快取未記錄 `client_email`、`project_id` 或 private key 指紋。

### 已重現結果

使用兩個不同 service account 呼叫 `getVertexToken()`：

- 第一個帳號取得 `token-1`
- 第二個帳號也直接收到 `token-1`
- OAuth fetch 總共只執行一次
- 第二個帳號的 private key 完全沒有被使用

若使用者切換 Vertex 帳號，約 58 分鐘內可能繼續使用舊身分 token，造成權限、計費專案或請求失敗混淆。

> `cloud-platform` OAuth scope 本身不等於擁有整個 GCP 專案權限；實際權限仍由 service account IAM roles 決定。

### 修復建議

- 快取 key 至少包含 `client_email + project_id`
- 更嚴謹可加入 private key 公鑰指紋
- API 設定儲存／切換 provider 時主動清除 Vertex token cache
- 使用 OAuth 回應的實際 `expires_in`，不要固定 3,500,000 ms

### 驗收條件

- [ ] 同一 service account 在有效期限內共用 token
- [ ] 不同 service account 一定重新換 token
- [ ] 修改 service account JSON 後不會沿用舊 token
- [ ] token 到期後會重新取得

## 6. 中優先：聊天匯入可載入外部追蹤圖片

### 證據位置

- `auris-vue/src/views/ChatRoomView.vue:78`
- `auris-vue/src/views/ChatRoomView.vue:1422-1439`

匯入會展開整個訊息物件：

```js
await dbPut('messages', { ...m, id: ..., charId });
```

畫面則直接載入：

```vue
<img v-if="m.image" :src="m.image">
```

不可信聊天 JSON 可放入外部 HTTPS tracking image。使用者開啟聊天時會向該主機發送請求，暴露 IP、時間與來源 origin。這不是 XSS。

### 修復建議

- 匯入時只接受預期格式，例如 `data:image/jpeg;base64,...`
- 非 data URL 圖片欄位直接移除
- 驗證 decoded MIME、base64 大小與像素上限

### 驗收條件

- [ ] 匯入含 `https://attacker.example/pixel` 的訊息不會產生網路請求
- [ ] 合法 Auris base64 圖片仍可正常還原
- [ ] 超大圖片會在寫入 IndexedDB 前被拒絕

## 7. 低優先：診斷資訊無法絕對保證不含對話

### 證據位置

- `auris-vue/src/services/llm.js:136-141`
- `auris-vue/src/services/diag.js:13-22`
- `auris-vue/src/services/diag.js:70-75`

LLM 服務商回傳的錯誤訊息會原樣寫入 ring buffer，最多 300 字。

### 已重現結果

假 API 回傳：

```text
provider echoed conversation: 我的私人對話
```

診斷 ring buffer 隨即包含相同文字。因此正常供應商通常不會回傳對話內容，但程式碼無法保證「絕不含」。

### 修復建議

- 診斷只記 provider、model、HTTP status、內部分類碼
- 不直接保存第三方 response message，或使用嚴格 allowlist／遮罩
- UI 文案改成「設計上不主動包含對話與金鑰；分享前仍請檢查」

## 8. 低優先：匯入缺少資源限制

以下匯入路徑都沒有檔案大小、資料筆數與字串長度上限：

- 全站備份
- 單角色備份
- 聊天 JSON
- 個人／角色頭像
- 聊天圖片

風險主要是使用者主動選取異常大檔後，造成主執行緒卡死、記憶體耗盡或 IndexedDB quota 用完。

建議為每種匯入格式設定：檔案大小、筆數、單則文字、base64 圖片與總資料量上限。

## 9. 低優先：部署與 CI 防禦縱深

### 正式站

- 有 HSTS
- CSP 透過 HTML meta 生效
- 沒有 HTTP `frame-ancestors`／`X-Frame-Options`
- 沒有 `X-Content-Type-Options`、Referrer Policy、Permissions Policy 等應用層 header
- CSP 含 `script-src 'unsafe-inline'`
- `connect-src` 與 `img-src` 允許所有 HTTPS，這與自訂 API／圖片需求有關，但降低 CSP 限制力

GitHub Pages 不方便提供自訂 response headers；若要完整 header 防線，需要 Cloudflare／自有 hosting／其他可控 CDN。

### CI/CD

- CI 使用 `npm install`，建議改為 `npm ci`
- deploy workflow 只 build，沒有再次 test/audit
- GitHub Actions 使用 `@v4`／`@v3` tag，未 pin commit SHA
- 專案內沒有 Dependabot config、CodeQL 或其他 SAST workflow

## 10. 工程健康與品質缺口

### 備份核心零測試

現有 `auris-vue/src/services/__tests__/backup.test.js` 只測：

- 備份提醒門檻
- 14 天週期
- snooze

沒有測試：

- `exportAllData()`
- `importAllData()`
- API key 排除與保留
- 缺 store
- 格式驗證
- transaction rollback
- quota／I/O 失敗
- 舊版 migration

這是本次高優先資料問題未被測試攔下的主要原因。

### 主題預讀使用舊 DB 名稱

- `auris-vue/index.html` 開啟 `AurisDB_v2`
- 實際 App DB 是 `auris`
- `SettingsView` 儲存主題時只寫 IndexedDB，沒有更新 `auris-theme` localStorage

可能造成：

- 冷啟動主題閃爍
- 建立無用的 `AurisDB_v2` 空資料庫
- 預讀程式無法讀到實際主題

### 其他品質項目

- `/lock` 目前是占位頁，不是實際 App Lock
- 沒有 lint script
- 沒有型別檢查
- 沒有 coverage gate
- 沒有正式 E2E suite
- 主 bundle 約 445 KB、gzip 約 146 KB；路由同步載入，後續可能繼續膨脹
- 本機 `node_modules` 有少量 extraneous 套件；改用 `npm ci` 可保持乾淨
